Aperçu

C'est quoi OpenFn ?#

OpenFn est une suite d'outils d'intégration de données, d'interopérabilité et d'automatisation du travail. Il est utilisés par les gouvernements, les ONG et les entreprises sociales, principalement dans les secteurs de la santé et de l'humanitaire.

Il comprend à la fois la plateforme OpenFn et OpenFn Integration Toolkit (boîte à outils d'intégration OpenFn) qui est libre et open-source.

:::astuce

En fait, c'est un logiciel qui facilite la tâche de ces organisations en leur permettant de connecter les différentes technologies qu'elles utilisent, de partager des données, de piloter des processus d'affaires critiques et de faire évoluer leurs interventions.

:::

C'est quoi ce site ?#

C'est le site de documentation unique pour tous les produits d'OpenFn. Vous êtes dans la section pour commencer.

Ici, vous apprendrez comment utiliser OpenFn pour construire des flux de travaux d'automatisation qui peuvent être aussi courts et simples ou aussi longs et complexes que vous le souhaitez. Vous apprendrez également tous les moyens que vous pouvez choisir pour déployer les flux d'automatisation que vous créez. Avant de plonger dans le démarrage rapide, commençons par quelques termes clés avant de continuer.

Termes clés#

Jobs#

OpenFn automation centers around jobs, which define the specific series of tasks or database actions OpenFn should perform. They can be set to be activated (triggered) at certain time intervals or when data matching specified criteria is received. You can think of jobs as a set of instructions you might give a data entry staff member (e.g., create a new Patient record in OpenMRS when a form containing a newly registered client is received from CommCare, export data to DHIS2 every week on Friday 11pm, send SMS with payment confirmation number when payment confirmation message is received etc.).

:::remarque

Les jobs sont entièrement configurables et réutilisables. They can also be chained together to create multi-step automation flows, two-way syncs. et pour maintenir la cohérence des données entre plusieurs applications (à l'aide des modèles Saga multi-applications). Vous pouvez en savoir plus sur la synchronisation bidirectionnelle ci-dessous.

:::remarque

Inbox#

Plus simplement, il s'agit d'une URL sécurisée qui peut recevoir des requêtes HTTP de n'importe où sur le Web. Chaque projet en a un sur OpenFn.

Messages#

Un message est un morceau de données qui est reçu par votre inbox. Cela peut trigger (voir ci-dessous) une exécution de job, et il contient le body (corps) et les headers (en-têtes) de la requête HTTP qui a été faite à votre inbox.

Runs#

Un run est chaque exécution individuelle d'un job. Imaginez qu'un job est configuré pour créer un nouveau patient dans OpenMRS chaque fois qu'un dossier est ouvert dans CommCare. Au cours de la semaine prochaine, si 5 cas sont ouverts dans CommCare, vous verrez 5 runs différents de ce job dans OpenFn. Si 4 runs sont réussis et qu'un a échoué, vous verrez 4 nouveaux patients dans OpenMRS, et votre administrateur système aura été averti que l'un de ces patients n'a pas pu être créé (ou la gestion plus robuste des erreurs que vous avez mise en place aura lieu.)

:::remarque

Sachez qu'il n'y a pas toujours une correspondance 1 à 1 entre les runs et les éléments réels avec lesquelles vous travaillez. Je pourrais définir un job qui récupère toutes les données d'évènements mises à jour à partir de DHIS2 depuis les 2 dernières semaines et les publier sur une carte publique en utilisant CartoDB. Ce job sera déclenché à des intervalles de temps spécifiés, toutes les 2 semaines dans ce cas, et après un mois, nous ne verrons que 2 runs dans OpenFn (soit un run toutes les deux semaines). Chaque run aura réussi ou échoué, et chacun peut avoir traité des milliers d'événements depuis DHIS2.

:::remarque

Job-states#

Given the many-to-one relationship between runs and messages A "message-job-state" (or simply a "job-state") is a calculation that can be useful for organizations that need to understand if a given message has eventually been handled successfully.

info

A job state is defined as the result ("success", "failure", or "in progress") of the last run (ordered by the time it was finished, rather than when it was inserted into the runs table) for a given message-job combination.

If two runs for the same message-job combination finished at the same time, it's ordered by their start time, and then finally by their primary key. In reality, since the same message-job combination can only be used to create a run once every 10 seconds, this will almost never occur.

Consider a message which should trigger both a case referral job and a payment job. Two runs will get created when the message arrives, with the referral succeeding and the payment failing. Navigating to the inbox, you'd see two "job-states" for that single message:

  1. Referral (success - run 1)
  2. Payment (failure - run 2)

If an administrator then made some sort of change, re-ran the failed payment job for that message, and this third run succeeded, you'd still only see 2 "job-states" in the inbox, but they'd both be successful:

  1. Referral (success - run 1)
  2. Payment (success - run 3)

Browsing to the receipt inspector would show all three runs for this single message.

Triggers#

Each job must be initiated with an event that acts as a trigger.

Triggers can be activated at a certain time (scheduled), when another job completes, when another job fails, or when a certain message is received.

Credentials#

A credential is used to authorize connection to a destination system (e.g., Salesforce username, password & login URL).

Adaptors#

Adaptors, are small, open-source modules that provide your jobs with the features they need to communicate with a particular system's API. Some examples are language-salesforce, language-dhis2, language-commcare, etc. There are ~50 active adaptors at the moment, and anyone is free to build or enhance them.
