Microservice

But#

OpenFn est utilisé par de nombreuses organisations sanitaires et humanitaires dans le monde entier pour développer leurs programmes grâce à l'interopérabilité en temps réel, l'intégration des systèmes et l'automatisation du flux de travail. OpenFn/microservice utilise la technologie open-core d'OpenFn : OpenFn/core, OpenFn/engine, et les différents adaptateurs OpenFn. Le but étant de créer des microservices autonomes qui peuvent être déployés sur n'importe quel matériel.

Cette approche de microservice permet de s'assurer que les gouvernements et les ONG ne sont jamais enfermés dans l'offre SaaS d'OpenFn, et peuvent porter leurs jobs existants, triggers et leurs credentials de OpenFn.org à leur propre infrastructure facilement.

Introduction#

Similaire à platform, OpenFn/microservice s'exécute sur les fichiers project.yaml. Cela signifie que lorsque des organisations ou des gouvernements ont une exigence de licence open-source, tous leurs jobs, credentials et configurations de projet peuvent être exportés depuis l'iPaaS d'OpenFn et utilisés pour créer un déploiement de microservices.

Bien que cette approche ne fournisse pas le front-end la plateforme OpenFn avec ses diverses fonctionnalités de gestion et de configuration de projet, elle est parfaite pour les groupes ayant une expérience de DevOps et 100% compatible avec la plateforme. Vous pouvez même construire et tester des projets entiers sur platform et ensuite exporter le fichier project.yaml à exécuter sur vos propres serveurs en utilisant microservice.

Cette approche de microservice fournit de la flexibilité aux gouvernements et aux ONG, de sorte qu'ils ne sont jamais enfermés dans l'offre de plateforme SaaS d'OpenFn. À tout moment, une organisation peut porter ses jobs, triggers et credentials existants d'OpenFn.org vers notre boîte à outils d'intégration FOSS, en utilisant leur propre infrastructure.

Prérequis#

La familiarité avec d'autres éléments de la boîte à outils d'intégration open source d'OpenFn est utile lorsqu'on envisage l'approche des microservices.

  • OpenFn/docs
  • OpenFn/engine
  • OpenFn/core
  • OpenFn/devtools

Docker opérationnel#

En supposant que vous avez .env et un exemple de projet dans le répertoire ./sample-project avec une spécification project.yaml :

docker-compose up

Vous pouvez configurer soit le fichier compose ou le .env ou exécuter le conteneur en utilisant docker run:

docker run -v <path-to-your-project-folder>:/home/microservice/<path-to-your-project-folder> \  --env-file <path-to-your-env-file> \  --network host \  openfn/microservice:v0.3.2

Guide de développement et de fonctionnement#

  • Cloner ce dépôt avec git clone git@github.com:OpenFn/microservice.git
  • Entrez le répertoire avec cd microservice
  • Installez les dépendances avec mix setup
  • Exécutez les tests avec mix test
  • Créez un répertoire de projet pour conserver les artefacts de votre projet mkdir sample-project
  • Créez une nouvelle spécification de projet avec cp project.yaml.example ./sample-project/project.yaml
  • Créez un fichier .env avec cp .env.example .env
  • Installez les adaptateurs nécessaires via npm install @openfn/language-http --prefix priv/openfn/runtime/node_modules --no-save --no-package-lock --global-style
  • Démarrez votre serveur de microservice avec env $(cat .env | grep -v "#" | xargs ) iex -S mix phx.server

Fonctionnement dans Docker#

  • Construire une image de docker avec docker build -t openfn/microservice:v0.3.0 .
  • Exécutez avec docker run command

Configuration de projet#

Vous pouvez configurer les jobs, triggers, credentials et les packs de langue utilisés dans votre microservice dans le fichier de configuration project.yaml.

Première configuration en utilisant l'exemple de configuration#

Le sample project configuration file décrit un exemple de configuration de projet pour vous aider à vous familiariser avec cette structure.

Microservice est configuré par défaut avec 4 exemples de jobs :

  1. job-1 est déclenché lorsqu'un message correspondant arrive à l'inbox (voir trigger-1).
  2. recurring-job est un job programmé pour s'exécuter toutes les minutes et est liée au trigger cron every-minute.
  3. flow-job et catch-job s'exécutent respectivement après success et failure de job-1.

Tous les jobs sont configurés avec le pack de langue openfn/language-common.

Dans l'exemple de configuration par défaut, un nouveau message posté à localhost:4000/inbox qui correspond à trigger-1 (c.à.d. le message contient "number":2). Il est accueilli avec un reçu d'accusé de réception asynchrone (HTTP 202 Les données acceptées et le traitement a commencé), ce qui déclenchera l'exécution de job-1.

Vous pouvez l'essayer avec l'extrait suivant :

curl -X POST -H "Content-Type: application/json" \ -d '{  "number":2,  "surveyId": 37479}' \ http://localhost:4000/inbox

Publier un message ne correspondant à aucun des triggers (par exemple “nombre”:3) entraîne également un accusé de réception mais ne déclenche aucun job.

Exemple de message pour ce scénario de non-concordance :

curl -X POST -H "Content-Type: application/json" \ -d '{  "number":3,  "surveyId": 37479}' \ http://localhost:4000/inbox

Les requêtes HTTP post faites à localhost:4000/inbox seront traitées par le Receiver, selon le credential, expression et adaptor défini dans le fichier YAML de la configuration du projet.

Les jobs time-based seront exécutés par Engine selon le credential, expression et adaptor définis dans votre fichier project.yaml.

Configuration depuis votre projet de plateforme OpenFn existant#

Si vous avez un projet configuré sur OpenFn, vous avez deux moyens d'exporter votre configuration sur la page Project Settings et d'exécuter votre projet en microservice.

  1. Si vous exportez en tant que project.yaml, vous pouvez télécharger vos paramètres au format yaml à partir de la page de téléchargement de votre projet de plateforme ou à partir d'un lien dans l'e-mail généré automatiquement envoyé à votre adresse. Vous pouvez insérer ce fichier dans votre environnement tel que configuré en utilisant le Development Up and Running Guide.

  2. Si vous exportez en tant que microservice.zip, vous obtiendrez votre dossier microservice prêt à fonctionner avec docker, contenant

  • un fichier de configuration docker-compose.yaml
  • un dossier du projet contenant project.yaml
  • Fichier .env avec les variables d'environnement par défaut pour docker
  • un fichier Readme

cd dans le dossier et exécutez le projet avec docker-compose up. Si vous n'avez pas l'image docker, elle sera automatiquement extraite de hub.docker.com.
